Encrypted Blob Vault - Spec v1

A write-only encrypted storage format where files emerge from indexed AEAD chunks inside random-looking blob files. The canonical state is carried by dual encrypted locator slots and append-safe index regions.

4096 Bfixed front matter
2locator slots per blob
40 BAEAD overhead per chunk

Front Matter And Commit Rule

Locator frame

24 B nonce + ciphertext + 16 B tag

Each 2032-byte slot is padded inside the ciphertext so the true payload size does not leak.

Payload

Generation, index pointer, IDs, keys, tail hash

The payload identifies the canonical index region and carries wrapped per-blob keys.

Reader choice

Highest valid generation wins

Both slots are decrypted. The valid slot with the highest monotonic generation is canonical.

Old slot remains canonicalThe existing generation keeps pointing at a valid prefix or prior layout.
Write inactive slotThe writer targets whichever slot is not currently canonical.
fsyncCommit only after the new slot is fully durable.
Reader decrypts bothPartial or invalid writes fail AEAD validation.
Pick highest validThe generation counter resolves the two-slot race.
Index becomes canonicalThe selected locator authorizes the index chain.
03

Crypto Key Relationships

vault_master_key

32-byte key provided by the app layer. It encrypts locator slots and wraps per-blob keys.

K_data

Random 32-byte key wrapped in the locator. Used as HKDF input for file keys.

K_file

HKDF(K_data, vault_id, "file/" || file_id). Encrypts all chunks for one file.

K_index

Random 32-byte key wrapped in the locator. Encrypts all index segments in that blob.

AEAD: XChaCha20-Poly1305 Hash: BLAKE2b-256 All multi-byte integers: little-endian All nonces and padding: CSPRNG
04

Chunk And Index Records

Chunk AEAD binds

File identity, sequence, and physical offset

AAD includes vault_id, blob_id, file_id, sequence_number, and chunk_offset_in_blob. Substitution, reordering, and transplantation fail validation.

Index segment AEAD binds

Blob, locator generation, and previous segment

AAD omits the segment byte offset so an index region can be relocated verbatim during Case 2 writes.

Write Protocol

Case 1 - append-only
Fit testC <= G and D <= T
Write chunkPlace it into the leading gap.
fsyncMake chunk durable before index claims it.
Append segmentWrite new encrypted index segment after index end.
fsyncMake the index extension durable.
New locatorBump generation and index length.
Case 2 - relocate index, then append
Shortfallmax(0, C + D - G)
Grow fileAppend random bytes to current EOF.
Copy indexMove index region byte-for-byte to new EOF.
fsyncRelocated index is durable.
New locatorGeneration authorizes relocated index.
Run Case 1Use the new large leading gap.
Crash invariant: no canonical index may reference a chunk that is missing from disk. Before Case 2's commit point, the old locator wins. After it, the relocated index is already fully written and internally consistent.

Recovery Path

Read front matterLoad bytes 0..4096 and the public salt.
Decrypt slotsValidate locator slot A and B with AEAD.
Select locatorHighest valid generation wins.
Walk indexValidate each segment and previous offset.
Check tailLast segment tag must match locator tail_hash.
Rebuild filesGroup by file_id, sort by sequence, verify completion.
If both locators fail: scan recovery is out of scope for v1 Forensic XChaCha-only extraction is unverified and must not be a normal read API
07

Concurrency Model

Writers

Exclusive advisory lock per blob

A writer must acquire the blob lock, re-read both locator slots, validate the full index under the lock, execute Case 1 or Case 2, fsync, then release.

Pre-lock state is only a hintPOSIX: flock LOCK_EXWindows: LockFileEx
Readers

No lock for normal reads

Committed chunks are immutable. Stale index views remain valid for known chunks. AEAD failures during a race are detectable and should trigger a front-matter reread and retry.

Shared lock optional for snapshotsNFS is not supported
08

Indistinguishability Properties

Observer without vault_master_key learns
  • Total file length.
  • Offset 0 contains a 32-byte random salt.
  • Offsets 32..4096 contain two random-looking locator regions.
  • Offsets 4096..EOF contain random-looking body bytes.
Observer cannot determine
  • Where data ends and the index begins.
  • How many chunks, files, or index segments exist.
  • Chunk or segment boundaries.
  • Whether two blobs share any files.
